# Stage Acceptance Review

Use this review before freezing any Engineering Stage.

## Engineering Review

- [ ] The stage answers one primary engineering question.
- [ ] The engineering guidance is technically sound.
- [ ] The stage does not introduce unnecessary future-stage complexity.
- [ ] Required decisions are clear.
- [ ] Expected artifacts are appropriate.

## Cognitive Review

- [ ] The page order reduces uncertainty.
- [ ] Concepts are introduced progressively.
- [ ] The engineer can identify what to do next.
- [ ] The stage avoids unnecessary repetition.
- [ ] The content supports confidence without reducing rigor.

## Navigation Review

- [ ] Every page identifies current stage and page.
- [ ] Previous and next links are present.
- [ ] No page ends without a next action.
- [ ] MkDocs navigation is complete.
- [ ] Cross-links are relative and likely to resolve.

## Evidence Review

- [ ] Required evidence is explicit.
- [ ] Repository locations are defined.
- [ ] Evidence supports downstream stages.
- [ ] AI-use evidence is addressed where relevant.
- [ ] Readiness gate checks evidence, not just activity.

## Publication Review

- [ ] No TODO markers.
- [ ] No placeholders.
- [ ] No draft disclaimers.
- [ ] No broken internal references identified.
- [ ] Terminology is consistent.
- [ ] Stage manifest is complete.
- [ ] Readiness gate is complete.

## Freeze Decision

A stage may be frozen only when all review categories pass.
